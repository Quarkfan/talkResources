AI 工程实践

现在很多人以为
接几个 API 就是 AI 工程了

这不是批评，而是一个被反复验证的判断——
套壳调 API 可以快速做出 demo，
但离真正的生产级 AI 系统，差得很远。

02 / 12
技术没过时

fine-tuning、训练、RAG、召回率
通用大模型太强，导致很多人误判了这些技术的价值

RAG 反而更重要了

模型回答错，往往不是推理错，而是前面的检索阶段已经错了——旧文档、过期 runbook、废弃配置、同名服务……上下文很脏，模型再强也救不回来。

小模型没消失

高频任务、低延迟、成本敏感、私有化部署、工单分类、日志归因——不是所有任务都值得调最强通用模型。

一句话区分

动态知识适合 RAG
稳定行为适合 fine-tuning

把企业知识灌进模型里通常是错的——知识在变，应该用 RAG；但输出格式、分类标准、工具调用习惯这些"行为"，微调效果更好。

Agent 的真相

Agent 不是"会调用工具"那么简单

低配 Agent

LLM
+
工具调用
+
Prompt
+
Workflow

能做 demo，能跑通流程。

生产级 Agent 还要什么

状态管理 权限系统 工具治理 上下文可信度 错误恢复 人工接管 审计日志 可回滚 风险分级 评估体系
Agent 的核心不是"能不能调 API"，而是能不能在真实系统中可靠地选择动作、执行动作、解释动作、恢复错误。

短期看，agent 最大的价值其实是连接——把多个系统之间的切换、搜索、复制封装成一个自然语言入口。但这不代表它真的"理解了企业"。

04 / 12
重新命名

"数字员工"这个词，掩盖了问题

"员工"暗示了什么

能理解组织目标
能承担职责
能判断优先级
能独立处理异常
能对结果负责

它实际是什么

一个会读上下文、会调用工具、会生成变更建议的自动化操作员。

员工有责任主体。Agent 没有责任主体。
"数字员工"这个词容易掩盖权限边界、审计责任、事故归因。

更准确的叫法：DevOps ChatOps Agent / 运维自动化编排器

05 / 12
最大风险

真正危险的不是一开始不好用

而是一开始太好用

早期试点——用户少、工具少、场景少、权限少、使用者懂系统——它会显得很聪明。
但这不能说明它具备大规模泛化能力。
类似低并发没问题，不代表高并发没问题。

06 / 12
复杂度爆炸

每接入一个平台，不是只多一个能力
而是增加了大量交叉失败模式

工具数量 × 上下文 × 用户角色 × 服务数量 × 环境数量 × 权限模型 × 任务类型 × 历史状态 × 风险等级

复杂度不是线性增加，而是组合爆炸。
接入更多工具同时扩大了正确动作空间和错误动作空间。

很多系统不是死在查询阶段
而是死在从"建议"滑向"执行"

08 / 12
治理思路

不能靠"模型更强"解决复杂度问题
必须靠系统治理

工具分级

只读查询可宽放开，MR 提交必须绑定 review，配置修改强审批，生产操作默认禁止白名单开放，删除/权限类极严控制。

上下文加质量标签

每段上下文带上来源、时间、环境、服务、版本、权限、可信等级。否则模型会把旧文档和新监控混在一起判断。

必须有拒绝机制

信息不足、风险过高、用户无权限、上下文冲突——如果一个 agent 永远"尽力回答、尽力执行"，它一定会出事。

复杂度压测

工具数量压测、上下文冲突压测、权限压测、并发状态压测——测试模糊指令是否越界。

09 / 12
能力分级

不要所有能力一起开放

L6 · 禁止 agent 执行的动作
L5 · 高风险动作强审批
L4 · 低风险自动执行
L3 · 生成 MR，人类 review
L2 · 生成建议
L1 · 查询总结

不要一个大 agent 包打天下——入口统一，内部拆成多个窄 agent 或确定性流程

10 / 12
评价标准

评价 agent 不能看"动作数量"
要看结果质量

看什么

成功率、采纳率、错误率、回滚率
人类修改比例、越权率
MTTR 是否下降
用户是否减少重复劳动

不看什么

调用了多少工具
提交了多少 MR
接入了多少平台
支持了多少任务
关键指标：每新增一个工具、平台、用户群后，原有任务成功率是否下降？新增能力导致旧能力退化 = 复杂度开始互相干扰。
11 / 12
总结

当前很多 AI 应用不是没价值

而是容易被过度命名、过度授权、过度扩张

API 套壳可以快速产生工具价值，但真正的生产级 AI 系统必须依赖数据、检索、评估、权限、状态、审计、回滚和人机协同。

12 / 12
结语

它不是不能做

而是不能被当成一个"越来越强的万能员工"来做

它必须被当成一个
高权限、不确定、需要治理的
自动化系统来做
