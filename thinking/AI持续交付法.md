# AI持续交付法

> **核心 Slogan**：AI工具化，工具AI化
>
> 面向有自有 IT 团队的制造企业，以两周为迭代周期，通过 IT 用 AI 开发小工具、全员提交业务痛点、投票决定开发优先级的方式，实现低成本、快速度的 AI 价值落地。

---

## 名字含义

| 词 | 含义 |
|------|------|
| **AI** | 大模型等 AI 能力，定位为 IT 的开发辅助工具 |
| **持续** | 每两周一个迭代节奏，持续收集痛点、持续交付工具、持续验证价值 |
| **交付** | 不是培训、不是咨询、不是方案，而是**可用的工具** |
| **法** | 方法论，有规则、有流程、有机制，不是临时做法 |

## Slogan 解读

**AI工具化，工具AI化**

- **AI工具化**：AI 不是业务人员的学习对象，不是培训课件，不是 PPT 里的概念——它是 IT 团队的开发工具。把 AI 从"要学的新技术"变成"用来干活的家伙"。
- **工具AI化**：用 AI 重构工具开发方式。原来 IT 开发一个小工具要 2 周，现在用 AI 辅助只要 1-2 天。不是做更少的事，是用更好的方式做事。

两句话合在一起，构成了方法论的完整闭环：**让 AI 变成工具（给 IT 用），让工具被 AI 构建（更快、更好、更多）。**

---

## 核心逻辑（一句话）

> 对于有自有 IT 团队的制造企业，AI 提效的第一站、也是最大的增量，永远在 IT 端的开发效率提升，而不是业务端的工具使用。

**不是让 100 个人学 AI，而是让一个人用 AI 给 100 个人做工具。**

## 三不原则

- **不教业务用 AI**——让 IT 用 AI 给业务做工具，业务一键操作
- **不做完美交付**——可用优先，有人用再优化，没人用就下架
- **不搞运动式推广**——不强制、不全员培训，用价值吸引自然传播

## 运转机制

```
痛点池（全员提交）→ IT 标注可行性 → 业务投票 → IT 两周交付 → 发布会展示 → 使用数据验证 → 循环
```

## 关键指标

| 维度 | 前 3 个月目标 |
|------|-------------|
| 交付工具数 | 10-12 个 |
| 活跃工具数（有人使用） | 5-6 个 |
| IT 节省工时 | 90-100 人天 |
| 业务节省工时 | 350-360 人天 |
| 总 ROI | 约 450 人天（保守打对折约 225 人天），零额外投入 |
